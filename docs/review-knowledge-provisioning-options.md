# レビュー知識(Agent Skills)提供方式の比較検討

本ドキュメントは、レビュアーにフレームワーク/言語/セキュリティ知識を与えている現行の
Agent Skills 方式を対象に、**より変更しやすく柔軟な知識提供方式**を検討したものである。
実装方針そのものではなく、複数アプローチを統一の観点で比較できる形に整理することが目的。

## 1. 背景

技術レビュー(`FrontendReviewer`)・セキュリティレビュー(`SecurityReviewer`)は、
`src/code_review_agent/skills/` 配下の Markdown ベースの Agent Skills
(`SKILL.md` + `references/*.md`、progressive disclosure)でフレームワーク/言語/OWASP知識を
与えている。この方式自体(知識をMarkdownで記述し、段階的にロードする設計)は機能しているが、
コードを追うと次の構造的な制約がある。

### 1.1 スキル束ねの配線がPythonコードに直書きされている

[`src/code_review_agent/skills/agent_skills_factory.py`](../src/code_review_agent/skills/agent_skills_factory.py)
の `AgentSkillType`(`StrEnum`)と `_build_frontend_review_skills()` /
`_build_web_security_review_skills()` が、「どのレビュアーにどのスキルパッケージ群を束ねるか」を
Pythonコードとして直書きしている。

```python
class AgentSkillType(StrEnum):
    NONE = ""
    FRONTEND_REVIEW = "frontend_review"
    WEB_SECURITY_REVIEW = "web_security_review"

def _build_frontend_review_skills() -> list[SkillSource]:
    return [
        Skill.from_file(_SKILLS_DIR / "reviewing-universal"),
        Skill.from_file(_SKILLS_DIR / "reviewing-languages"),
        Skill.from_file(_SKILLS_DIR / "reviewing-frameworks"),
        Skill.from_file(_SKILLS_DIR / "reviewing-metaframeworks"),
    ]
```

各レビュアー([`agents/reviewers/frontend.py`](../src/code_review_agent/agents/reviewers/frontend.py)、
[`agents/reviewers/security.py`](../src/code_review_agent/agents/reviewers/security.py))は
`skill_type: ClassVar[AgentSkillType]` でこの束を選択する。新しいスキルパッケージ
(例: 新フレームワーク、Spring Boot 等の新スタック向け知識)を追加するたびに、
**enum値の追加 + builder関数の追加 + レビュアークラスの `skill_type` 配線**という
コード変更が必要になる。

### 1.2 コンテンツ変更もコードと同じCIゲートを通る

スキル本体の `.md` ファイルの内容変更(チェックリストの文言修正など)も、Pythonコードの変更と
全く同じCIゲート([`.github/workflows/ci.yaml`](../.github/workflows/ci.yaml)の
`uv run pytest --cov-fail-under=75`、`ruff check`、`ruff format --check`)と
CONTRIBUTING.md の Spec-Driven + TDD プロセスを通る。Markdownの文言修正であっても
コード変更と同等のレビュー・CI負荷がかかる。

### 1.3 デプロイはビルド時焼き込み、ホットリロードの仕組みがない

[`deploy/pod.yaml`](../deploy/pod.yaml) の通り、デプロイは単一Pod
(`podman kube play` 想定、Chainguard distrolessイメージ、シェル/curlなし)構成であり、
スキル内容はDockerイメージにビルド時に焼き込まれる。`deploy/configmap.yaml` は環境変数の
`envFrom` のみでファイル配信のボリュームマウントは組まれていないため、内容を直しても
イメージの再ビルド・再デプロイなしに反映することはできない。

### 1.4 今回のヒアリングで確認した優先課題

上記の構造的制約のうち、特に次の2点が優先課題として確認された(以降の比較・推奨パスはこの2点を軸にする)。

1. **新規パッケージ追加時のコード変更**(1.1) — スキルを追加・差し替えるたびに
   `agent_skills_factory.py` の enum/builder関数を書き換える必要がある。
2. **反映までのリードタイム**(1.3) — 内容を直したら再デプロイなしで本番レビューに
   即時反映したい。

## 2. 統一比較観点

以降のアプローチ比較は、すべて次の9つの観点で評価する。

| ID | 観点 | 定義 |
|----|------|------|
| C1 | 変更容易性 | 誰が編集できるか、どんなレビュー/承認ゲートを通るか(フルTDD+カバレッジPR / 軽量content限定レビュー / ゲートなし) |
| C2 | 反映までのリードタイム | コンテンツ変更が実際のレビューで使われるまでの時間(即時 〜 再デプロイ待ち) |
| C3 | 新規パッケージ追加時のコード結合度 | 新しい知識パッケージ(新フレームワーク等)の追加が、Python/レジストリのコード変更を要するか |
| C4 | バージョン管理・ロールバック・監査性 | 変更履歴の追跡、任意バージョンへのロールバック、誰が何をいつ変えたかの監査可能性 |
| C5 | コンテンツ品質担保の仕組み | 誤った/古い/悪意ある知識が混入するのを防ぐ機構(lint・スキーマ検証・人的レビュー等)の有無 |
| C6 | 実行時コスト・レイテンシ | その方式自体がレビュー1回あたりに追加するトークン・API呼び出し・レイテンシ |
| C7 | 精度 と 網羅性/鮮度 のトレードオフ | 厳選されたチェックリストの高精度・低ノイズ さと、公式ドキュメント等の広い網羅性・最新性 のどちらに寄るか |
| C8 | 運用複雑性・新規インフラの要否 | 新しいサービス・ストレージ・デプロイ形態が必要か(現状: 単一Pod、ボリュームマウントなし) |
| C9 | 複数リポジトリ/複数チームでの知識共有可能性 | 他リポジトリ・他チームが同じ知識を再利用したり独立にバージョン管理したりできるか |

## 3. 比較対象アプローチ

### ベースライン: 現状の Agent Skills

- **内容**: `AgentSkillType` enum + `_build_*_skills()`、コンテンツはDockerイメージに焼き込み、
  LLMが `file_read` で progressive disclosure。
- **評価**:

| 観点 | 評価 |
|------|------|
| C1 | 本リポジトリへのPR、CONTRIBUTING.mdのフルTDD+カバレッジゲート |
| C2 | イメージ再ビルド+再デプロイが必要 |
| C3 | **高結合** — 新パッケージは enum値 + builder関数 + レビュアー配線が必要 |
| C4 | git履歴のみ(本リポジトリと一体、ロールバックは`git revert`で容易) |
| C5 | ruff/pytestのみ(コンテンツ内容自体を検証するlintはない) |
| C6 | なし(`file_read`呼び出しは既存のスキル起動上限内) |
| C7 | 高精度(厳選済み)、鮮度は手動更新に依存 |
| C8 | なし(現状そのもの) |
| C9 | なし(本リポジトリ内に閉じている) |

### A. 宣言的マニフェスト化 + 自動ディスカバリー

- **このコードベースでの変更点**: `AgentSkillType` enum と `_build_*_skills()` を廃止し、
  (a) `skills/` 配下で `SKILL.md` を持つディレクトリを `Path.glob("*/SKILL.md")` で自動検出、
  (b) レビュアー種別 → スキルパッケージ一覧 を対応付ける小さなマニフェスト
  (`skills/bundles.yaml` 等)を追加。`create_agent_skills(bundle_name: str)` はマニフェストを
  読んで `Skill.from_file()` を並べるだけになる。レビュアークラスは
  `skill_type: ClassVar[AgentSkillType]` を `skill_bundle: ClassVar[str]` 相当に変更。
  マニフェストの整合性(参照先ディレクトリの実在)はテストで検証する。
- **工数感**: 小(半日〜1日。`agent_skills_factory.py` の書き換え + 既存テスト
  [`tests/skills/test_agent_skills_factory.py`](../tests/skills/test_agent_skills_factory.py) の更新のみ、新規インフラなし)。
- **評価**:

| 観点 | 評価 |
|------|------|
| C1 | ベースラインと同じ(本リポジトリPR、フルCIゲート) |
| C2 | ベースラインと同じ(イメージ焼き込みのまま、再デプロイ要) |
| C3 | **低〜なし** — 新パッケージはディレクトリ追加 + マニフェスト1行、Pythonコード変更ゼロ |
| C4 | ベースラインと同じ(git履歴) |
| C5 | ベースライン + マニフェスト整合性テスト(参照切れをCIで検知) |
| C6 | 変化なし |
| C7 | 変化なし |
| C8 | なし(最も低コストな案) |
| C9 | 変化なし |

**今回の最優先課題(1.4-1)を最小コストで直接解消する案。**

### B. 知識コンテンツの別リポジトリ/別配布物への分離

- **このコードベースでの変更点**: `skills/reviewing-*` と `skills/web-security-review/` を
  別リポジトリ(例: `code-review-knowledge`)へ移し、git submodule あるいはバージョン付き
  配布物(OCIアーティファクト等)としてDockerビルド時に取り込む。`agent_skills_factory.py` は
  `_SKILLS_DIR` の解決先を変えるだけの小変更で済む。本リポジトリのCIはコンテンツpathを対象外にし、
  分離先リポジトリには markdown-lint 等の軽量CIを別途用意する。
- **工数感**: 中(2〜4日。別リポジトリのガバナンス設計、Dockerfileのfetchステージ追加、
  分離先リポジトリの軽量CI構築を含む)。
- **評価**:

| 観点 | 評価 |
|------|------|
| C1 | **改善** — コンテンツPRは軽量ゲート(markdown-lint等)のみ、非エンジニアにも開放しやすい |
| C2 | 変わらず(依然としてビルド+デプロイが必要) |
| C3 | 低(パッケージ追加は分離先リポジトリでのcontentのみ + 本体側のバージョンpin更新) |
| C4 | **改善** — 独立したsemver/タグ管理、ロールバックはpin変更のみ |
| C5 | 新規に軽量ゲート(markdown-lint、参照整合性チェック)を作り込む必要あり(自動では付いてこない) |
| C6 | 変化なし |
| C7 | 変化なし(引き続き厳選コンテンツ) |
| C8 | 中(別リポジトリの維持、Dockerfileのfetchステップ、認証等の追加依存) |
| C9 | **改善** — 他リポジトリ/他エージェントが同じコンテンツを独立にpin・再利用できる |

### C. 実行時ホットリロード(外部ストレージ配信)

- **このコードベースでの変更点**: コンテンツを外部ストレージ(ConfigMap/オブジェクトストレージ等)
  から配信し、`agent_skills_factory.py` の `_SKILLS_DIR` を環境変数駆動のパスにする。
- **重要な補正**: 一般的なKubernetesであれば「ConfigMapをボリュームマウントすればkubeletが
  自動的に内容を同期する」ためコンテナ再起動なしに反映できるが、本リポジトリの実際のデプロイは
  `deploy/pod.yaml` に見る通り **単一Pod(`podman kube play` 想定)・ボリュームマウントなし**
  という構成である。したがって、この方式を採用する場合は「K8sネイティブなConfigMap自動同期」を
  そのまま前提にはできず、(a) ボリュームマウント自体の追加、または (b) オブジェクトストレージから
  定期的にpullするsyncの仕組み(sidecarまたはアプリ内バックグラウンドタスク)を新規に実装する
  必要がある。加えて、バージョン管理・ロールバック・品質ゲートも自前で設計しないと、
  誤った内容が数分でレビューに反映されてしまうリスクがある。
- **工数感**: 中〜大(ボリュームマウント方式のみなら1〜2日+デプロイ変更、
  オブジェクトストレージ同期まで作り込むなら1週間程度、新規の失敗モード対応含む)。
- **評価**:

| 観点 | 評価 |
|------|------|
| C1 | Bと組み合わせる場合はBのC1に準ずる。単独では本リポジトリPR起点のまま |
| C2 | **最良** — 理論上は再ビルド・再デプロイ不要で分単位の反映が可能 |
| C3 | Aと組み合わせることが前提(単独では解決しない) |
| C4 | **弱い** — 素朴なストレージ配信にはバージョン管理・ロールバックの仕組みがなく、別途設計が必要 |
| C5 | **弱い** — 本リポジトリのCIゲートを経由しないため、配信経路自体に品質ゲートを作り込まない限り無防備 |
| C6 | レビュー実行自体への追加コストは軽微(コンテンツは既にローカルにある前提) |
| C7 | 変化なし |
| C8 | **最も高い(Markdownベース案の中で)** — 新規デプロイ構成変更、配信・同期の仕組み、新しい障害モードの監視が必要 |
| C9 | ストレージの持ち方次第(namespaceを分ければマルチテナント化も可能だが追加設計が要る) |

「反映までのリードタイム」(1.4-2)に最も効くが、**現状の単一Pod構成を前提にすると相応の追加インフラ投資が要る**。

### D. RAG(公式ドキュメント等のベクトル検索)

- **このコードベースでの変更点**: ベクトルDB・埋め込みパイプライン・新規ツールモジュール
  (`src/code_review_agent/tools/` に `github_mcp.py` 相当のもの)を新規に構築し、
  `LLMReviewAgent` に `uses_rag: ClassVar[bool]` のようなトグルを追加する。
- **工数感**: 大(2週間以上。ベクトルストア、インジェストパイプライン、鮮度維持の運用まで
  必要で、既存コードベースに完全に新しいサブシステムを追加することになる)。
- **評価**:

| 観点 | 評価 |
|------|------|
| C1 | 「編集」ではなく「再インジェスト」という別種の運用になる |
| C2 | インジェスト頻度次第(準リアルタイム〜バッチ) |
| C3 | パイプライン構築後は低いが、最初の構築自体が大規模なコード変更 |
| C4 | 弱い(埋め込み/チャンクのロールバックは自明ではない) |
| C5 | **最も弱い** — 人手によるキュレーションがなく、もっともらしいが誤った/古いスニペットを拾うリスクがある |
| C6 | **最も高い** — 検索・埋め込み呼び出しとコンテキストトークンが毎回増える |
| C7 | 網羅性・鮮度は最良、精度は最も低い |
| C8 | 高い(ベクトルDB・インジェストジョブ・鮮度監視という新規サブシステム) |
| C9 | 中央ホスト化すれば共有可能だが追加設計が要る |

既存のシステムプロンプト(`reviewing-universal/system_prompt.md`)が明示的に
「無関係な知識を読み込ませない」progressive disclosure設計を志向しているため、
**チェックリストの置き換えではなく補助的な用途に留めるのが妥当**。

### E. リモートMCP知識サーバー

- **このコードベースでの変更点**: 別途MCPサーバーを立て、
  `src/code_review_agent/tools/knowledge_mcp.py` のような新規ツール(`github_mcp.py`と同型)
  を追加し、`uses_knowledge_mcp: ClassVar[bool]` トグルを `LLMReviewAgent` に追加する。
- **工数感**: 本リポジトリ側は中(1〜2日、`github_mcp.py`のパターンを踏襲)だが、
  知識サーバー自体の構築・運用(認証、コンテンツストア、可用性、独自リリースプロセス)は
  別個の大規模な取り組みになる。
- **評価**:

| 観点 | 評価 |
|------|------|
| C1 | **最良** — サーバー側のコンテンツストアで独自の編集/承認フローを持てる。非エンジニアも本リポジトリに触れず貢献可能 |
| C2 | **最良** — サーバー側の更新が次のツール呼び出しで即座に反映される |
| C3 | ほぼゼロ(新パッケージはサーバー側の追加のみ) |
| C4 | サーバー側の設計次第(本リポジトリには何も付いてこない) |
| C5 | サーバー側に品質ゲートを作り込めば現状より強くできるが、それ自体が新規の設計・実装 |
| C6 | スキル呼び出しごとにネットワークラウンドトリップが追加(既存の呼び出し上限内なら軽微だが、サーバー障害が新しい単一障害点になる) |
| C7 | サーバー側のコンテンツ方針次第(中立) |
| C8 | **最も高い(組織的な意味で)** — 常時稼働サービスの新規構築・運用・SLA維持が必要 |
| C9 | **最良** — 複数リポジトリ/複数チームでの共有を前提に設計されている |

複数リポジトリでの知識共有や非エンジニアの直接貢献が本当に必要になった場合の選択肢。
**今回ヒアリングした2つの優先課題(1.4)には直接効かず、投資対効果が見えにくい。**

### F. A + B のハイブリッド(パス限定CIによる軽量ゲート)

- **このコードベースでの変更点**: Aを実施した上で、
  [`.github/workflows/container-security-scan.yml`](../.github/workflows/container-security-scan.yml)
  で既に使われている `paths:` フィルタと同じパターンで、`skills/**/*.md` のみを変更するPRには
  `pytest`/`pyright`/カバレッジのフルゲートをスキップし、markdown-lint + 「SKILL.mdが参照する
  ファイルが実在するか」の整合性チェックのみを課す軽量ワークフローを追加する。
  別リポジトリを立てずに、Bの「content変更を軽量ゲートに」という利点の一部だけを安価に得る。
- **工数感**: 中(A: 0.5〜1日 + 新規ワークフロー・整合性チェックスクリプト: 1〜2日、
  合計で1週間程度。新規インフラ・新規リポジトリなし)。
- **評価**: 基本的にAの評価を踏襲しつつ、C1(content変更は軽量ゲートのみ)とC5
  (整合性チェックという新規の軽量ゲート)が改善する。C4は現状維持(独立したsemverは持たない)。

## 5. 推奨パス

今回ヒアリングで確認した優先課題は「(1) 新規パッケージ追加時のコード変更」と
「(2) 反映までのリードタイム」の2点(1.4節)。この2点を軸に、次の段階的な進め方を推奨する。

1. **今すぐ: A(宣言的マニフェスト化 + 自動ディスカバリー)**
   優先課題(1)を最小工数(新規インフラなし、既存コードの小規模リファクタのみ)で直接解消する。
   B/C/E のどれを将来採用するとしても、配線がマニフェスト化されていることは無駄にならない。
2. **次に検討: F(パス限定の軽量CI) → 効果を見てから C(ホットリロード配信)を判断**
   優先課題(2)を本気で解消するならCが最も効くが、現状の単一Pod・ボリュームマウントなし構成を
   前提にすると、配信の仕組み・バージョン管理・品質ゲートを新規に作り込む必要があり
   運用複雑性(C8)が最も高い。まずFで「content変更が重いCIを通る」不満を安価に緩和しつつ、
   「本当に分単位の即時反映が必要か、次回デプロイまでの待ちで十分か」を実運用で見極めてから、
   Cへの投資要否を判断することを提案する。
3. **非優先: D(RAG)・E(リモートMCP知識サーバー)は今回不採用**
   D/Eが強く効く課題(網羅性・鮮度の限界、複数チームでの知識共有)は今回の優先課題(1.4)に
   含まれていない。将来これらのニーズが顕在化した場合の選択肢として記録するに留める。

## 6. オープンクエスチョン

実装フェーズに進む前に、次の点をユーザーと合意しておく必要がある。

- Cを本格的に採用する場合、配信基盤(ConfigMap+ボリュームマウント、オブジェクトストレージ、
  その他)としてどれを選ぶか。現状のPodman単一Pod構成をどこまで変更してよいか。
- Cやリモート配信を採用する場合のコンテンツ品質ゲート(C5)を誰がどう設計するか
  (現状はコードと同じCIが暗黙の品質担保になっている)。
- 将来 Spring Boot 等の新スタックを追加する際、Aのマニフェスト形式(YAML/JSON、
  reviewer→スキルパッケージ一覧)で十分か、それとも観点(perspective)ごとに異なる
  マニフェスト構造が必要か。
- B/Eのように知識を本リポジトリの外に出す場合、[`docs/review-agents-design.md`](review-agents-design.md)
  5章の「AgentSkillsは実装済み」という記述、および評価パイプライン
  ([`evaluation/EVALUATION_PLAN.md`](../evaluation/EVALUATION_PLAN.md))が参照するテスト前提を
  どう更新するか。
