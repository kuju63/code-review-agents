# ADR-0004: MCPクライアントのセッション共有(レビュアー間)

- Status: Proposed(未実装・レビュー待ち)
- Date: 2026-07-13
- Related: Issue #115, [docs/adr/0003-github-mcp-startup-retry-strategy.md](0003-github-mcp-startup-retry-strategy.md)

## Context

Issue #115は評価パイプライン実行中にGitHub MCP接続エラー(`background thread did not
start in 30 seconds` / `Connection to the MCP server was closed` / 空ボディのJSONパース
失敗)が多発した問題について、原因分析の結果「1. リトライを入れる」「2. MCPクライアントの
使い回し」の2方針で対応することを既に合意している。方針1は
[ADR-0003](0003-github-mcp-startup-retry-strategy.md)として確定し、PR #116でマージ済みである。
ADR-0003のDecision項6は「『MCPクライアントの使い回し』(Issue #115 item2、セッション共有)は
本ADRのスコープ外とし、別Issueとして切り出す」と明記しており、本ADRはその方針2の対象範囲と
セッションのライフサイクル管理方式を確定させるものである。

ADR-0003は「1PRあたり最大3本のMCP接続が立つ(PR情報収集1本+レビュー実行時の並列レビュアー分)」
という数値を輻輳の根拠として挙げているが、本ADR作成にあたりアプリケーション全体の処理の流れを
精査した結果、この3本は必ずしも同時に存在するわけではないことが判明した。1件のPRレビューは
「PR情報収集」→「複数レビュアーによる並列レビュー」→「最終判定の合成」という3つの処理段階を
順に実行する構成になっており、PR情報収集の段階が完了して接続を終えてから、次の並列レビューの
段階が始まる。したがって、PR情報収集の接続と並列レビューの接続は時間的に重ならず、実際に
輻輳(同時多重)の当事者なのは、並列レビューの段階で同時に起動されるレビュアー間の接続
(現状は同時に2本)のみである。この事実は「どの範囲で接続を共有するか」を検討する上での
前提になる。

また、本プロジェクトが依存している`strands-agents`パッケージ(`pyproject.toml`で`>=1.45.0`を
指定)が提供するMCPクライアント実装(`strands.tools.mcp.mcp_client.MCPClient`、`strands.tools.
tool_provider.ToolProvider`インターフェースの実装)は、1つの接続を複数の利用者(consumer)で
共有し、最後の利用者が離脱した時点で自動的に接続を終了する仕組み(`add_consumer`/
`remove_consumer`)をフレームワーク側の機能として備えている。`MCPClient`クラスのdocstringにも
「同一コネクションを複数のツール呼び出しで再利用してレイテンシを削減するためのコンテキスト
マネージャパターン」である旨が明記されており、複数の呼び出し元で同一コネクションを再利用する
利用形態を想定した設計であることが読み取れる。このため、セッション共有そのものは技術的に
選択可能な選択肢である。一方で、この仕組みを使わず個々の利用者が独自に接続の開始・終了を
管理する現状の実装から移行する場合、接続の起動処理を複数の経路から重複して行おうとすると
起動処理そのものが競合して失敗しうる、という点はライフサイクル管理方式を検討する上で
考慮する必要がある。

## 検討事項

1. 共有範囲: MCP接続をどの単位(並列レビュー実行の内部のみ/PR情報収集を含むPRレビュー全体/
   サーバープロセス全体)で共有するか。
2. 共有セッションのライフサイクル管理: 共有する接続の開始・終了を誰が・どのタイミングで
   管理するか。

## 検討内容

### 検討事項1: 共有範囲(共有単位)

| 選択肢 | メリット | デメリット |
|---|---|---|
| A. 並列レビュー実行の内部でのみ共有する | 輻輳の当事者である「並列に実行されるレビュアー群」に直接効く。PR情報収集と並列レビューという既存の処理段階の境界を変えずに済み、影響範囲を並列レビュー実行の内部だけに閉じられる | PR情報収集側の接続数(1本)は変わらないため、1PRレビュー全体で見た接続確立回数の削減効果は限定的 |
| B. PR情報収集を含めPRレビュー全体で共有する | 1PRレビュー全体で見た接続確立回数(≒起動試行の総数)がさらに1本分減る | PR情報収集と並列レビューは処理段階として時間的に重ならないため、輻輳(同時多重)の解消にはほとんど寄与しない。一方で、性質の異なる2つの処理段階をまたいで1つの接続を受け渡す構成が新たに必要になり、全体の見通しが悪くなる |
| C. サーバープロセス全体でシングルトンとして共有する(PR横断) | 輻輳を最も強く抑えられる(理論上、常に接続は1本のみ) | 1本の接続に問題が起きた場合、その瞬間に処理中の全てのPRレビューに影響が及ぶ(障害の影響範囲が最大になる)。Issueで合意した「同一PRレビュー内での共有」という前提を超え、PRをまたいだ接続の生存管理という新たな検討事項を持ち込むことになる |

### 検討事項2: 共有セッションのライフサイクル管理

| 選択肢 | メリット | デメリット |
|---|---|---|
| A. 利用者(レビュアー)からの参照がなくなったら自動的に終了する、参照カウント方式 | 利用者の数の増減に応じて接続の生死が自動的に管理され、利用者側が個別に終了処理を意識する必要がない。フレームワーク側が標準で備える仕組みであるため、独自の状態管理を新たに作り込む必要がない | 「最後の利用者がいつ終了するか」が暗黙的になるため、想定外のタイミングで接続が終了する、あるいは終了されない、といった状態を追跡しづらくなる |
| B. 単一の管理者が接続の開始・終了を明示的に管理する自前の仕組み | 開始・終了のタイミングが明示的で、いつ接続が確立・終了するかを追跡しやすい | 参照数の管理(カウンタ・排他制御等)を独自に設計・維持する必要があり、例外発生時の後始末漏れなど、自前で作り込む範囲がAより広くなる |
| C. 複数接続をプールとして用意し、利用者に割り当てる方式 | 1本の接続に処理が集中するリスクを分散できる。将来、同時に稼働するレビュアー数が大きく増えた場合に有効な選択肢になりうる | 現状の利用者数(同時に2)に対しては明らかに過剰な仕組みであり、プールサイズの決定や空き接続待ちの扱いなど、運用上の意思決定・管理対象が増える |

## 検討結果

### 検討事項1: 共有範囲 → A(並列レビュー実行の内部でのみ共有)を採用

輻輳の当事者は並列レビュー実行段階のレビュアー間のみであり(Context参照)、PR情報収集は
時間的に重ならないためそもそも輻輳の当事者ではない。この前提に立つと、Bが得られる追加の
メリット(接続確立回数のさらなる削減)は輻輳解消には結びつかず、得られる効果に対して
処理段階をまたぐ構成の複雑化という代償の方が大きい。Cは輻輳抑制の効果こそ最大だが、
1本の接続障害が同時進行中の全PRレビューに波及するという影響範囲の大きさが、Issueが
前提とする「同一PRレビュー内での共有」というスコープに見合わない。Aは輻輳の当事者に
過不足なく対応し、既存の処理段階の境界を変えないため影響範囲も限定的であることから、
効果とリスクの釣り合いが最も良い選択肢として採用する。

PR情報収集は本ADRの対象外とすることを明示する。将来、レビュアー数の増加などにより
PR情報収集側の接続も輻輳の一因になると判明した場合は、改めて別Issueとして検討する。

### 検討事項2: 共有セッションのライフサイクル管理 → A(参照カウント方式)を採用

Bは、参照カウントによる自動終了という仕組みをフレームワークが既に備えているにもかかわらず、
それを使わずに独自の管理機構を一から作ることになり、実装量とバグの起こりやすさの両面で
Aに劣る。Cは検討事項1で共有範囲をレビュアー間のみに絞った結果、同時に管理すべき利用者数が
少数(現状2)にとどまるため、複数接続を用意して分散させるほどの規模ではなく、複雑さに
見合う効果が薄い。Aはフレームワークが標準で備える仕組みをそのまま利用できるため実装・
保守の負担が最小であり、デメリットである「終了タイミングの追跡しづらさ」は、共有範囲を
検討事項1でレビュアー間のみに限定したことで管理対象自体が小規模になり、実務上許容できる
範囲に収まると判断する。

## Decision

1. MCP接続の共有範囲は、並列に実行されるレビュアー群の内部のみとし、PR情報収集は対象外とする。
2. 接続の終了は参照カウント方式(利用者がいなくなった時点で自動的に終了する仕組み)を採用し、
   フレームワークが標準で備える仕組みをそのまま利用する。
3. レビュアー群の並列実行を統括する側は、この参照カウントに対して自らも「利用者」として
   振る舞う: 全レビュアーの並列実行を始める前に接続を確立して自身の参照を保持することで
   接続の寿命(生存期間)を固定し、各レビュアーは通常の利用者として参照を持つ。全レビュアーの
   処理が完了した後、統括側が自身の参照を最後に解放することで、参照カウントの仕組みにより
   接続が自動的に終了する。個々のレビュアーが自分自身の判断で接続を終了させる現状の振る舞いは、
   共有接続を使う経路では行わないよう改める。
4. MCPを利用しないレビュアーや、レビュアーを単体で利用する既存の利用形態のために、共有接続が
   提供されない場合は現行通り個別に接続を確立・終了する振る舞いをフォールバックとして維持する。
5. ADR-0003で決定した起動リトライは、接続確立の主体が変わることを踏まえ、本ADR後の接続確立
   経路に対しても適用されるようにする。

## Consequences

- 1PRレビューあたりの同時MCP接続数がレビュアー数(現状2)から1に削減され、Issue #115の
  輻輳の真因を直接軽減する。
- PR情報収集は対象外のため、そちらの接続数(1本)は変わらない。輻輳解消の効果は「並列に
  実行されるレビュアー間」に限られる。
- 複数のレビュアーが同一の接続を介してツール呼び出しを行うことになるため、呼び出しは
  共有接続側で順次処理される形になり、レビュアー数が増えるほど1本あたりの待ち時間が
  増加しうる。GitHub MCPへの呼び出しは読み取り専用でI/Oバウンドな性質であるため、
  現状のレビュアー数(2)では許容範囲と判断する。
- 将来レビュアーが増えた場合、この1本の共有接続がボトルネックになる可能性がある。その場合は
  検討したが今回は見送ったセッションプール(検討事項2の選択肢C)を再検討する。
- ADR-0003の起動リトライは、本ADR後の接続確立経路に対して適用されるよう、着手時に適用箇所を
  再確認する必要がある。
