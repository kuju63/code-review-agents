## 7. 残課題の切り分け（実体に基づく）

修正前の主因「ツールを一度も呼ばない（データ取得の欠落）」は**解消**した。
残る誤差は性質が異なり、2つに分かれる。

### 7.1 構造化時の忠実性（copy-fidelity）の問題

run 1 / 8 / 11 は **ファイルパスが完全に正しい（Recall 1.0〜0.8）= ツールで実ファイル一覧を取得できている**
にもかかわらず、**Title と Label のみ言い換え・創作**している。

| run | 取得ファイル | 生成 Title（正解と不一致） | 生成 Label |
|-----|------|------|------|
| 1 | 5件すべて正解 | `Update CircularProgress/LinearProgress warnings` | `['bug', 'enhancement']` |
| 8 | 5件すべて正解 | `Fix: Handle warning flags in CircularProgress` | `[]` |
| 11 | 4/5 正解 | `Fix: Fix missing type definitions in mui/material-ui` | `['bug', 'type:feat']` |

→ データは会話文脈に存在する（ファイルが正しいことが証拠）のに、構造化ステップで
タイトル・ラベルを**忠実に転記せず要約・創作している**。これは取得の問題ではなく、
gemma-4-e4b（約4B級）の**忠実性（指示追従）の弱さ**。レポート6.4「設計→再計測→モデル比較」の
順序に従い、よりツール追従・忠実性の強いモデル（例: gemma-4-12b-qat）での再計測が次の一手。

### 7.2 ツールループの長時間化と環境失敗（9/20）

実行時間が平均 16.6s → 312.6s（最大 1039s = 約17分）に激増した。
これは小型モデルがツールループ内で多数の試行を繰り返すため。20回計測の後半で：

- run 12: 437秒のループ後に `peer_keepalive_timeout`（LM Link 接続エラー）
- run 13–20: LM Studio がモデルを再ロードできず `insufficient system resources`（約3秒で即時失敗）

これらは**コード不具合ではなく環境（LM Studio のメモリ枯渇）要因**だが、
「案Aは小型ローカルモデルでツールループが長時間化・暴走しやすい」という**運用上の実体課題**を示す。
レポート6.2 の検証ガード（toolResult=0 や不整合での失敗扱い・リトライ）に加え、
**ツールループのステップ上限・タイムアウト**の導入を次の改善候補とする。

## 8. 結論

- **案A（ツールループ→構造化の2段分離）＋ file一覧対処（SYSTEM_PROMPT強化）により、
  PR Info Collector の正確性は 0% 水準から大幅に改善した**
  （Title 完全一致 0%→73%、File F1 0.00→0.69、Label 完全一致 0%→73%、幻覚ファイル 50→4）。
  レポート5.3 で特定した根本原因（`structured_output()` 単独使用でツール未呼び出し）が
  実体として解消されたことが、toolUse 発生と実ファイルパスの取得で裏付けられた。
- ただし **N=11（20回中9回が LM Studio のメモリ枯渇で環境失敗）** であり、統計の母数が縮小している点は留保。
- File F1・Label が暫定基準（0.8 / 0.9）に未達なのは、もはやデータ取得ではなく
  **小型モデルの構造化時の忠実性**が主因（§7.1）。次は順序どおり**モデル比較**（6.4）へ進むのが正しい。
- 併せて**ツールループの長時間化・暴走の抑制**（ステップ上限/タイムアウト、検証ガード）が運用上の必須改善（§7.2）。
